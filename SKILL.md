# CV 论文创新点挖掘

帮助科研用户从多篇计算机视觉论文中，挖掘出**可落地、能解决真实存在的问题、改进后能提升效果指标**的创新框架，并做撞车检查避免重复造轮子。

## 这个技能要交付的核心价值

用户做这件事的真实目的是**找到一个能写成论文、能跑出涨点结果的创新点**。所以判断输出好坏的标准，按重要性排序是：

1. **可落地、能实现** —— 提案要具体到模块/机制层面，说清楚"动哪里、怎么动"，而不是"可以引入注意力机制"这种正确的废话。
2. **解决真实存在的问题** —— 每个创新点必须挂钩到某篇论文里**确实存在**的局限，而不是为创新而创新。
3. **改进后能提升指标** —— 要从机制层面解释"为什么这样改能涨点"，并给出可验证的实验设计。
4. **不撞车** —— 提醒用户哪些想法可能已经被做过，给出差异化建议。

写报告时始终带着这四条自检。如果一个提案说不清它解决什么真实问题、为什么能涨点，就不要写进去——宁缺毋滥。

## 工作流程

### 第 1 步：确定阅读方案（按论文数量自适应）

先数一下用户喂入了几篇 PDF，再决定读法。论文越多，越要先做减法，把精力放在最相关的几篇上：

- **1–4 篇**：逐篇精读全文，重点是方法（method）和实验（experiments）两节，弄清每篇的机制细节和报告指标。
- **5–10 篇**：每篇先读摘要、引言、方法概述，判断与用户意图的相关度；对最相关的 3–5 篇精读方法与实验，其余做要点提取即可。
- **10 篇以上**：先用摘要+图表做一轮快速筛选，按"是否同一子任务、是否有可迁移模块"挑出核心子集（一般 4–6 篇）精读，其余仅作为背景引用。

把你采用的阅读方案在报告开头简短说明，让用户知道哪些是精读、哪些是略读。

### 第 2 步：确认研究方向并加载领域常识

用户喂入的论文属于某个 CV 子方向。识别方向后，**只加载对应那一份**领域知识参考文件，不要全部读进来（省上下文、也更聚焦）：

- 目标检测 → 读 `references/detection.md`
- 图像/实例/全景分割 → 读 `references/segmentation.md`
- 目标跟踪（SOT/MOT 等） → 读 `references/tracking.md`
- 通用骨干网络、训练技巧、跨方向共性知识 → 读 `references/common.md`（多数情况下都值得搭配主方向一起读）

如果方向不明确或跨多个方向，可以加载多份，但要克制。这些文件提供该方向的经典 baseline、常用数据集与评价指标，用来判断"现有 SOTA 大概在什么水平""新提案该跟谁比"。

### 第 3 步：逐篇解析（创新点 + 仍存在的问题）

对每篇精读的论文，提取：
- **核心任务**：解决什么 CV 问题。
- **创新点**：1–3 条，**具体到模块或机制**（例如"用 X 注意力替代了 Y 的全局匹配，降低了计算量"），而不是泛泛而谈。
- **关键方法**：整体 pipeline 一句话讲清。
- **报告指标**：在什么数据集上、什么指标、达到多少。这是后续判断"能否涨点"的锚点，务必记下。
- **仍存在的问题/局限**：这是挖创新点的金矿。要找**具体、可改进**的局限（如"在小目标/遮挡/长时跟踪上掉点""依赖大量标注""推理速度慢"），避免"泛化性有待提高"这种没法下手的空话。优先参考论文自己在 limitation/conclusion 里承认的不足，再结合你的判断补充。

### 第 4 步（可选）：联网搜索相关解决方案

是否联网由用户决定。如果用户没明确说，且任务涉及"找解决某问题的最新工作"或"撞车检查需要查证"，**主动问一句**是否要联网搜索。

- **联网时**：用 WebSearch / WebFetch 针对第 3 步归纳出的关键问题，检索近期相关工作，记录其思路与出处（标题、会议/期刊、年份、链接）。这些将作为模块迁移的素材和撞车检查的依据。
- **不联网时**：基于已有知识给出可能的解决思路，但**凡是具体的论文引用都标注"⚠️需人工核实出处"**——因为脱离联网时模型可能记错或杜撰文献。这一点对科研用户极其重要，宁可标注不确定，也不要给出看似精确实则可能虚构的引用。

### 第 5 步：整理出一个完整的新模型（而不是罗列模块迁移）

这是技能的核心产出。用户要的不是"把 A 的模块搬到 B"这种菜单式罗列，而是**一个可以直接拿去做的完整模型**：给出明确的 **baseline + 一组（建议 2–3 个）具体改动**。其中至少有一条是**技能自己提炼的原创改动**（不能全是纯迁移），且这条原创必须**有参考佐证**——能指出它建立在哪篇论文的哪个观察/机制之上，让创新可信而非空想。

每个创新点的来源可以是以下任意一种，**只要理论上能提升指标就可以纳入**：
- **对 baseline 自身模块的小改**（例如换一种更平滑的权重、把固定值改成自适应）；
- **从用户喂入的其他论文迁移过来的模块**；
- **联网检索到的、可改进 baseline 的外部模块/技巧**。

组织方式：
- **选定 baseline**：从精读论文里挑一个最合适的现有强方法做底座，说清为什么选它。
- **列出 2–3 条具体改动**：每条落到"动哪个模块、怎么动、接口怎么对齐"，具体到能照着写代码的程度，而不是"引入 XX 机制"。最好让这几条改动**互相协同、讲成一个故事**（如一条改进另一条所依赖的信号），而不是互不相关的拼盘。
- **标出每条的来源与性质**：是小改 baseline / 迁移自某篇 / 联网外部模块，以及哪条是原创核心；原创那条写清"它建立在 [某篇] 的 [某观察] 之上"。优先去做**子领域里还空白**的组合（结合第 4 步联网结果判断哪里挤、哪里空），并尽量让方案**落地友好**（如免重训可插入、零额外推理开销）。
- **涨点锚点**：对照第 3 步记录的 baseline 原始指标，说清这个改动预期在哪个数据集/属性上提升、机制上为什么。
- **实验设计**：baseline / 数据集 / 指标 / 关键消融。

判断这一步好坏：把模型描述给一个该方向的研究生看，他能不能直接照着搭起来、并说出"这个新点子建立在什么已有工作之上"。说不清就再具体化。

### 第 6 步：撞车检查（novelty check）——优先联网

对新模型的每个改动（尤其那条原创）评估新颖性。**撞车检查应尽量联网**，因为这一步最怕拍脑袋：
- 分别检索"底层 primitive 是否已存在（哪怕在别的领域）"和"在本子领域里是否已被做过"——这两件事结论常常不同（通用领域有、本领域没有，正是机会所在）。
- 列出最接近的 1–2 篇、说明相似点，给出**风险等级（高/中/低）+ 理由 + 差异化建议**。
- 未联网时对具体文献引用标注"⚠️需人工核实"，并主动建议用户允许联网再下结论。

### 第 7 步：给出方案推荐程度

不要把所有想法平铺。用**推荐程度（⭐⭐⭐ / ⭐⭐ / ⭐）+ 一句理由**排序，明确告诉用户先做哪个、哪些不建议碰（通常是已经很挤的方向）。

## 报告结构（精简优先）

默认输出要**简短、直给**。除非用户要详细逐篇报告，否则按下面这个精简骨架（术语可保留英文）：

```markdown
## 推荐模型：[名字]（暂名）
**Baseline：** [选哪个现有方法 + 一句为什么]
**修改建议（2–3 条，注明来源：小改 baseline / 迁移自某篇 / 联网外部模块）**
- ① [核心/原创改动] —— 动哪里、怎么动；标明这是原创点，建立在 [某篇] 的 [某观察] 之上
- ② [迁移或外部改动] —— 从 [某篇/某外部工作] 迁移什么、接口怎么对齐
- ③ [小改或协同改动] —— 与 ①/② 怎么配合、为什么理论上能涨点
**涨点锚点：** 对照 baseline 原指标，预期在哪提升、机制原因
**实验设计：** baseline / 数据集 / 指标 / 消融

## 撞车检查（联网）
- 通用领域是否已有该 primitive；本子领域是否已被做过；最接近的工作
- 风险等级 + 差异化建议

## 各方向推荐程度
| 程度 | 方向 | 理由 |
（⭐⭐⭐ 推荐 / ⭐⭐ 谨慎 / ⭐ 不推荐）
```

只有当用户明确要"完整逐篇解析"时，才在前面补一节"论文逐篇解析"（核心任务 / 创新点 / 关键方法 / 报告指标 / 仍存在的问题）。否则逐篇细节折叠进推理过程，不占报告篇幅。

## 几个容易翻车的地方

- **别堆砌空泛建议**。"引入注意力""加多尺度融合""用更强的 backbone"这类谁都能说的话，对科研用户没价值。每条建议都要落到"哪篇的哪个模块、迁到哪里、为什么"。
- **别杜撰文献**。未联网时凡涉及具体论文名/作者/数值的引用都要标注不确定。科研用户被一条虚构引用坑过一次就不会再信任这个工具。
- **指标要可比**。提出涨点时，要锚定到第一节记录的原始指标和对应数据集，不要凭空说"能提升 5 个点"。
- **宁缺毋滥**。两三个扎实、能落地的提案，远好过十个听起来很炫但没法实现的点子。
